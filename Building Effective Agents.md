# Building Effective Agents

https://www.anthropic.com/engineering/building-effective-agents

## What are Agents?

### Agent（智能体）

- LLM 自主组织整个执行过程
- 在运行时动态决定下一步、是否调用工具
- 决策流程由模型**内部逻辑控制**

### Workflow（工作流）

- 代码路径由开发者提前硬编码
- LLM 只在部分步骤中被调用
- 模型不决定整体流程，仅依赖程序逻辑

它们都属于 **agentic system**，但是否是真正的“自主智能体”，取决于是否由 LLM 主导控制。

---

## When (and when not) to use agents

使用 LLM 构建应用时，应优先寻找最简单的方案，仅在必要时增加复杂度。

很多情况下，其实不需要 agent。  
因为代理系统通常会以**延迟和成本**为代价，换取更高性能，因此需要权衡是否值得。

- Workflow：适合流程明确、需要稳定性和一致性的任务
- Agent：适合需要灵活性和模型自主决策的任务

而对于很多应用，仅通过检索 + 上下文示例优化单次 LLM 调用，往往已经足够。

---

## When and how to use frameworks

常见 agent 框架：

- Claude Agent SDK
- AWS Strands
- Vellum 等

### 优点

- 简化调用 LLM、工具解析、链式调用等底层工作
- 更容易快速上手

### 缺点

- 隐藏内部实现，调试困难
- 容易过早引入不必要复杂度

建议先直接调用 LLM API，理解底层机制后，再决定是否使用框架。

# Building Blocks, Workflows, and Agents

## （1）增强型 LLM（Augmented LLM）

代理系统的核心构建模块。

通常包括：

- Retrieval（检索）
- Tools（工具）
- Memory（记忆）

现代模型可以主动：

- 生成搜索查询
- 选择工具
- 决定保留哪些信息

重点是：  根据具体业务场景，对这些能力进行定制，并提供清晰、易用的接口。

---

## （2）Prompt Chaining

流程：

1. 将任务拆分为多个子步骤
2. 每一步调用 LLM
3. 将上一步输出传给下一步

目标：

- 用更高延迟换取更高准确率
- 让每次 LLM 调用只处理简单任务

适用场景：

- 可清晰拆解的固定任务流程
- 如：写作 → 翻译

---

## （3）Routing（路由）

先对输入分类，再路由到不同处理流程。

适用场景：

- 存在明显类别
- 每类任务适合单独处理
- 分类能够稳定执行

例子：

- 简单问题 → 小模型
- 复杂问题 → 强模型

---

## （4）Parallelization（并行）

多个任务同时执行，再统一汇总结果。

两种常见形式：

- **Sectioning**：拆分子任务并行执行
- **Voting**：同一任务运行多次获取不同结果

适用场景：

- 子任务可独立执行
- 需要多个视角提高结果置信度

例子：

- 内容安全审核
- 多维度判断违规风险

---

## （5）Orchestrator-Workers（协调器-工作者）

中央 LLM 动态拆解任务，并分配给多个 Worker LLM。

特点：

- 子任务不是预定义的
- 由协调器根据输入动态生成

适用场景：

- 无法提前确定步骤的复杂任务
- 如大型代码修改任务

例子：

- 多来源信息搜索与分析

---

## （6）Evaluator-Optimizer（评估-优化器）

一个 LLM 生成结果，另一个 LLM 负责评估与反馈，形成循环优化。

适用场景：

- 存在明确评估标准
- 迭代优化能带来明显收益

两个典型特征：

1. 人类反馈能明显改善结果
2. LLM 自身也能提供有效反馈

例子：

- 翻译优化
- 文案改写

---

# Agent 自主执行逻辑

流程：**明确目标 → 自主规划 → 循环执行 → 获取反馈 → 动态调整**

Agent 在调用工具时会获得真实环境反馈，并基于反馈调整后续计划。

任务会在：

- 完成目标
- 或达到停止条件（如循环上限）

后结束。

适用场景：

- 开放性问题
- 无法预测步骤数量
- 无法硬编码固定路径

但 Agent 的问题在于：

- 成本更高
- 错误可能累积

因此需要：

- 沙箱环境测试
- Guardrails（防护机制）

# Conclusion

在 LLM 领域取得成功，并不在于构建最复杂的系统，而在于**构建最适合自己需求的系统**。从简单提示（prompt）开始，通过全面评估优化，然后仅在简单方案无法满足需求时再引入多步骤代理系统。

在实现agent时，我们遵循三个核心原则：

1. **保持设计简洁**——保持agent设计尽量简单明了。
2. **优先透明性**——明确展示agent的规划步骤。
3. **精心设计agent-计算机接口（ACI）**——通过详尽的工具文档和测试确保接口可靠。

框架可以帮助快速入门，但在进入生产环境时，不要犹豫降低抽象层级，使用基础组件进行构建。
