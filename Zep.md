# ZEP: A Temporal Knowledge Graph Architecture for Agent Memory

设计了一种基于时间感知知识图谱（Temporal Knowledge Graph）的记忆架构，让 AI 能像人一样形成长期记忆。

## 一、研究背景

虽然 GPT 等大模型能力很强，但仍然受到几个限制：

**（1）上下文窗口有限**

模型一次只能读取有限长度的信息。

模型无法把全部历史对话都放进 Context Window。

**（2）RAG 主要针对静态知识**

适用于：

- 产品说明书
- 公司文档
- 知识库

但不适用于：

- 用户最近换了工作
- 用户偏好发生变化
- 企业实时数据更新

因为这些信息是动态变化的。

**（3）Agent需要长期记忆**

需要记忆“随时间变化”。

而普通向量数据库很难处理这种关系。

---

## 二、研究方法

不要只存文本，而是构建一个带时间信息的知识图谱。

核心组件叫：Graphiti

Graphiti 是一个动态时间知识图谱引擎。

它能够：

- 持续接收新信息
- 自动更新事实
- 保留历史记录
- 支持时间推理

Zep 就建立在 Graphiti 之上。

### 系统架构

作者设计了三层知识图谱结构：

**第一层：Episode Layer（情节记忆）**

保存原始数据。

例如：

```
2025-01-01
用户：我在Google工作
```

原文直接存储。

即：“我经历过什么事情”。

**第二层：Entity Layer（语义记忆）**

从对话中提取实体和关系。

例如：

原文：

```
我在Google工作
```

转换成：

```
User—WORKS_FOR—Google
```

形成知识图谱中的节点和边。

**第三层：Community Layer（社区层）**

进一步把相关实体聚类。

例如：

```
Android Chrome Gemini
```

聚成：

```
Google Ecosystem
```

形成更高层次的概念总结。

这样检索时既能看细节，也能看全局。

### 时间知识图谱设计

普通知识图谱的问题

假设用户说：

2024年我在微软工作

后来2025年我加入OpenAI了

传统图谱可能变成：

```
User → WORKS_FOR → Microsoft
User → WORKS_FOR → OpenAI
```

两条关系同时存在。

系统不知道哪条是最新的。

Graphiti的解决方案

每条关系都带时间。

例如：

```
WORKS_FOR(Microsoft)
有效期:2024-01-01~2025-01-01
```

系统既知道：

- 当前事实
- 历史事实

也知道：什么时候发生变化

### 工作流程

**阶段1：知识图谱构建**

**Step1 实体抽取**

从消息中提取实体（llm）：

例如：

```
我喜欢苹果手机
```

提取：

```
我 苹果手机
```

使用 LLM 完成抽取。

**Step2 实体消歧**

例如：

```
Apple 苹果公司 苹果
```

可能是同一个实体。

系统：

- 向量检索
- 全文检索
- LLM判断

共同完成实体合并。

**Step3 关系抽取**

从实体之间提取关系。

形成知识图谱边。

注：先实体抽取去重再关系抽取去重

**Step4 时间抽取**

识别时间表达。

例如：

```
两周前我换工作了
```

结合消息时间：

```
2025-01-15
```

推导：

```
有效开始时间=2025-01-01
```

并记录进知识图谱。

**阶段2：检索**

当用户提问时：Zep 不只是向量检索。它结合三种搜索方式：

1.向量检索

Cosine Similarity：找语义相似内容。

2.BM25全文检索

找关键词匹配。

3.图搜索（BFS）

沿知识图谱扩散。

```
OpenAI
   ↓
GPT-4
   ↓
ChatGPT
```

即使用户没提 ChatGPT，也能找到相关信息。

这是普通向量数据库做不到的。

三个检索从不同角度找候选，再统一排序筛选，最后只保留 Top-K。

候选实体怎么找？

论文不是直接让 LLM 看整个图谱。成本太高。

而是先做向量搜索、全文搜索BM25，然后把候选结果给 LLM。

| 模块        | 是否用LLM |
| --------- | ------ |
| 实体抽取      | ✅      |
| 实体消歧      | ✅      |
| 关系抽取      | ✅      |
| 关系去重      | ✅      |
| 时间抽取      | ✅      |
| 冲突检测/失效判断 | ✅      |
| 社区摘要生成    | ✅      |
| 向量检索      | ❌      |
| BM25检索    | ❌      |
| BFS图搜索    | ❌      |
| 最终回答生成    | ✅      |

---

### 工作流程

1.接收数据

系统持续接收用户对话、文本或企业业务数据，并保存为 **Episode（原始记录）**。

2.构建时间知识图谱

从数据中自动提取：

- **实体（Entity）**：人、公司、地点等
- **关系（Fact）**：实体之间的联系
- **时间信息（Time）**：关系何时开始、何时失效

例如：

```
User --WORKS_FOR--> OpenAI
有效期：2025-01-01 ~ Now
```

如果用户后来换工作，系统会自动更新旧关系的失效时间。

3.检索相关记忆

当用户提问时，系统不会直接读取全部历史记录，而是通过：

- 向量检索（语义相似）
- BM25关键词检索
- 图谱搜索（关联关系）

快速找到最相关的事实和实体。

4.生成回答

将检索到的少量关键信息提供给 LLM，由 LLM 生成最终回答。

## 三、实验设计

**实验1：DMR Benchmark**

DMR（Deep Memory Retrieval）是 MemGPT 提出的记忆评测集。

包含：

- 500组多轮对话
- 每组5个会话
- 约60条消息

任务：

```
给定历史对话回答一个记忆问题
```

例如：

```
用户最喜欢什么饮料？
```

**实验流程**

1. 导入所有对话
2. 构建Graphiti图谱
3. 检索最相关实体和事实
4. 提供给GPT回答
5. 与标准答案比较

**实验结果**

| 方法                   | 准确率       |
| -------------------- | --------- |
| Conversation Summary | 78.6%     |
| MemGPT               | 93.4%     |
| Full Conversation    | 94.4%     |
| Zep                  | **94.8%** |

### 实验2：LongMemEval

作者认为 DMR 太简单。

因为：60条消息全部能放进GPT上下文。

无法体现长期记忆能力。

于是使用：**LongMemEval**

特点：

- 平均 115,000 tokens
- 超长对话
- 更接近真实企业场景

包含六类问题

**结果**

使用 GPT-4o：

| 方法           | 准确率       |
| ------------ | --------- |
| Full Context | 60.2%     |
| Zep          | **71.2%** |

提升：**18.5%**。

延迟表现

| 方法           | 平均响应时间 |
| ------------ | ------ |
| Full Context | 28.9秒  |
| Zep          | 2.58秒  |

速度提升约：**90%**。

## 四、总结

### 创新点

时间感知知识图谱：记录事实 + 时间。普通向量数据库没有的

动态更新而不是重建图谱：每天都有新对话。持续写入、持续更新

### 不足

大量依赖 LLM：整个写入链路几乎都依赖 LLM。成本高、错误传播(一旦实体识别错误，错误会逐层传播)

没有真正证明图谱一定优于向量库：只比较Full Context VS Zep

社区层（Community Layer）贡献不明确：没有消融实验——没有回答去掉Community会怎样？

### Q&A

**1.这篇论文主要解决什么问题？**

论文主要解决 Agent 的长期记忆问题。传统 RAG 更适合静态文档检索，而智能体需要记住不断变化的用户信息和业务数据，例如工作变动、偏好变化等。Zep 通过构建时间感知知识图谱，使 Agent 不仅能记住事实，还能记录事实何时生效、何时失效，从而支持长期记忆和时间推理。

**2.为什么作者要用知识图谱？**

因为知识图谱天然适合表示实体之间的关系。这种结构比向量更容易进行关联推理。同时图谱支持 BFS 搜索，可以发现多跳关系，这对于长期记忆和复杂推理更有优势。

**3.如果用户信息发生变化怎么办？**

系统会执行 Edge Invalidation（边失效机制）。当新的事实与旧事实冲突时，Graphiti 不会删除旧事实，而是为旧边记录失效时间，并让新边生效。因此系统既保留历史信息，又能维护当前状态。

**4.检索阶段为什么同时使用 BM25、向量检索和 BFS？**

三种方法关注的信息不同：

- BM25：关键词匹配
- Embedding：语义相似
- BFS：图结构关联

三者互补，提高召回率。系统先获得候选结果，再通过 reranker 重排序，而不是简单取交集。

**5.LLM 在系统中主要负责什么？**

LLM 主要用于知识图谱构建，包括：

- 实体抽取
- 实体去重
- 关系抽取
- 关系去重
- 时间抽取
- 冲突检测

检索阶段主要依靠图数据库和检索算法完成，因此查询速度较快。
